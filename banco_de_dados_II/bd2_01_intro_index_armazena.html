Comparao das organizaes de arquivos
Operaes consideradas
A Varredura
B Pesquisa (igualdade)
C Pesquisa (intervalo)
D Insero de um registro

Modelo de custo
B: Nmero de pginas

rvores B+)
D: Tempo de E/S de uma pgina (operao mais
cara)
C: Tempo para processar um registro
H: Tempo para computar uma funo hash

IFMG 24
ndices e sintonizao de desempenho
ndices podem ser criados para tornar consultas mais ecientes
Porm, certas alteraes nos dados podem causar atualizaes de vrios ndices
Isso pode levar a problemas de desempenho no sistema
O ideal haver um equilbrio, ou seja, os ndices so ferramentas poderosas que
devem ser usados com cautela
IFMG 25
Avaliaes somente de ndice
Quando todos os campos buscados em uma consulta fazem parte da chave do ndice,
tal consulta pode ser avaliada usando apenas o ndice
No necessrio buscar os dados no arquivo
Normalmente, o ndice menor do que o arquivo, isso faz com que a consulta seja
avaliada mais rapidamente
IFMG 26
Chaves de pesquisas compostas
As chaves de pesquisas compostas possuem mais de um atributo
Tais chaves podem suportar faixas maiores de consultas
Alm disto, elas aumentam as chances de avaliaes somente de ndice
Por outro lado, o ndice precisa ser atualizado se ocorrer qualquer alterao em um dos
campos que compem a chave
Outra desvantagem o espao a mais ocupado pelo ndice
IFMG 27
ndices no PostgreSQL
A criao de ndices no PostgreSQL com a instruo CREATE INDEX
CREATE INDEX [nome] ON tabela [ USING mtodo]
(coluna_1, ..., coluna_n);
Os principais mtodos so:
btree: rvore B (mtodo padro, nico que suporta unicidade)
hash: Hash (no funciona para mltiplas colunas)
